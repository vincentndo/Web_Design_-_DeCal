Part 1: A Quick Review of Paths

Before we dive into our file structure, it's important that we understand relative and absolute paths well.

(Quick tip: the word "directory" means the same thing as the word "folder" you'd use to refer to folders on your computer. We use those two terms interchangeably.)

A path is a piece of text that points our computer or browser to a specific file or folder. It tells the computer where to look to find the desired file. Let's look at a few examples of absolute and relative paths:

The / (forward slash) at the beginning of /hello/world/hi.html denotes that it's an absolute path. If a path is absolute, it tells the computer to start looking for the desired file at "root" (base-level folder) of the computer's hard drive. In this case, we look in the hello folder inside of our root folder, and then the world folder inside of our hello folder, where we should find hi.html.

If we instead write hello/world/hi.html, without the leading forward slash, we're using a relative path. If a path is relative, we start looking in the current directory--that is, the directory (or folder) that our current HTML file is in. To put this in more concrete terms:

  • With the example given, we first look in the current folder for a folder called hello, and then within that folder, look for a folder called world, within which we find hi.html.
  • Let's say that our current directory is called project1, and is inside of another directory called wddprojects. If we were to move project1 outside of the wddprojects folder, all of the links on our website--all of our relative paths--would still be intact! Nothing would break, because all of our paths are written as relative paths, meaning they tell you the location of files relative to what folder the current HTML file is in.

We can also refer to the current or previous directory using relative paths. We write the current directory as . (single dot), and the parent directory as .. (two dots). The directory's grandparent is then referred to as ../.., its great grandparent is ../../.., and so on. Some examples of this being used with our previous example:

  • ./hello/world/hi.html points to the same file we pointed to in the previous example.
  • ./ points to our project1 folder.
  • ../ points to our wddprojects folder.
  • ../project1/hello/world/hi.html points to the same file we pointed to in the first bullet point, though you'd seldom use a path like this, since it's rather repetitive (the entire ../project1/ can be truncated to ./ or just empty space).

Why do we usually prefer relative to absolute paths?

  • On a webpage, we might not always know our absolute path, but we always are able to find things from the perspective of our current directory (relative).
  • Relative paths are much shorter than their absolute equivalents, which makes them much easier to write out.
  • Our paths are no longer dependent on any folder other than the root folder of our project! In other words, if we move our project folder into another folder (or onto another web domain), nothing changes--our website will not break. It's safer!

Part 2: Linking Paths

As you continue to learn more about web development and design, you'll find that it's helpful to have more than one webpage in a website. Each HTML file serves as a single page[1] in your website, like a home page, an about page, or a page for viewing blog articles. To link to another page, use an <a href="..."> anchor tag! You've seen anchor tags before in previous homework assignments and labs. Let's explore an example:

<nav>
    <a href="about.html">About</a>
    <a href="#contact">Contact</a>
    <a href="./help.html">Help</a>
    <a href="./articles/1.html">First Article</a>
</nav>

Consider the navigation links given above in a file called index.html. (Those pages themselves are not necessarily standard for a navbar.)

  • In the first link, we link to about.html, which is a relative path to a file called about.html that's in the same directory as our index.html (the file from which we're linking).
  • In our next link, we link to #contact, which points to the element with the id contact on the current webpage (index.html). The page will scroll down to make the #contact element visible.
  • In the third link, we link to a file called help.html in our current directory using another relative path (in a slightly different style).
  • In our fourth link, we link to a file called 1.html inside of an articles directory inside of our current directory.

Part 5: Different Project, Different Paths?

While the file structure that we've introduced here is simple, streamlined, and flexible enough for other projects, you might wonder whether other file structures are used, or when you might want to use a different structure. The answer is yes, and often! The file structure used is dependent on what type of project you're building, the team with which you're building it, and the tools and frameworks you're using to build it.

For projects that use vanilla HTML, CSS, and JavaScript, the file structure that we use in WDD is sufficient and effective. It enforces a good separation of concerns, and you can easily add more files to it while staying organized. It's also convention to use our file structure within the scope of all WDD projects, which makes it easier for others in WDD (like instructors, TAs, and group members for your group project) to understand how your code works.

But WDD's structure isn't the only effective structure for web development. It works best in the scope of this course, and projects similar to those that we assign. Info on other file structures is far out of the scope of this course, but you can find more in our footnotes[3] if you're curious. Just know that there are many, many ways to organize your files for a given project.

With so many options, how do you know which file structure to choose for your project? In WDD, we recommend that you use our suggested file structure. But beyond WDD, here are some factors to take into account:

  • What file structure will make my application most scalable? That is, which file structure is easiest to add to and expand in an organized way as I continue to build, rebuild, and rebuild again my application over the years (trust us, you will!)?
  • What file structure makes the most sense to my team and me? What file structure is organized enough and intuitive for us to use? What kinds of roles are those on my team serving? If our team is interdisciplinary (we have some members who are not developers, but perhaps are responsible for design, branding and marketing, research, or some artistic pursuit), then what file structure will best help us to integrate the work of other creatives on the team with our digital product?
  • What stack are we using? (A stack is all of the technology you're using for a project, including frameworks, languages, etc.)
  • How have others organized projects like the one that I'm working on? Are there popular patterns[2] that exist already? Why are those patterns popular?

Realistically, your file structure will not make or break your project. You can always change what you've created. That being said, having an organized file structure from the get-go will make your workflow as a developer and designer much simpler.

Don't worry if you find this overwhelming--you don't need to know all of this for the class! But we thought we'd give you a good look at things you might get to explore in the future as you continue to build projects of your own.

[1]: Single-Page Websites

Although each HTML page traditionally serves as one page, with the power of JavaScript, we can build websites whose entire content spans only one HTML page! Consider the WDD site, for example: the entire landing page, despite having many scrollable sections, is written as one webpage. Some webapps also employ this sort of functionality. Other websites use an entirely different system of pages based on JavaScript rather than HTML, where different HTML elements are rendered inside of one parent div on the same page based on routing that a JavaScript framework handles. Some websites also do this to create transitions or animations between their pages, though the same can be achieved with a multiple HTML file structure as well by using certain other JavaScript frameworks.

[2]: Popular Patterns

Patterns exist for a lot more than just file structure--you'll also find patterns for how to name your CSS classes (e.g. BEM), what stack to use for certain types of projects (a few examples here), and more. If it's been done before by enough people, there'll likely be a pattern for it.

There are also tools that exist to help you maintain patterns of development! Yeoman is a build tool for easily creating new parts of a project while following a preset structure (based on frameworks, etc.). A lot of text editors and other tools can check the style of your code (i.e., the way you've written a program, like whether you used tabs or spaces, how you separate out your braces and brackets per line, and more).

On large projects, you might even use tools that can automatically build your project for you! When you use languages or frameworks that have to be processed before they're uploaded to the web (like TypeScript, Babel or newer versions of ECMAScript, JSX, or SCSS, among others), you can create developer pipelines that make your workflow simpler and more efficient (typing out one command or clicking on one button rather than many in order to have your files ready to go). Using tools like these might also change your file structure--you might separate all of your files out into a src/ folder, while files used by tools for your workflow exist at the root directory of your project. Examples of build tools that work like this are NPM, Gulp, Grunt, and Webpack, none of which you'll need for the course, but offer a lot of power later along the line, and are fascinating from a technological standpoint.

[3]: File Structures Far Down Along The Line

If you're curious about file structures that JavaScript-heavy projects use, read on! The following is very far out of the scope of the class.

Some projects are built around a structure of components, or reusable parts of a website that update based on some state. Projects like that often separate their components into different folders, which contain JavaScript files that represent those components and CSS files used to style each component.

Some projects are structured entirely around a given JavaScript framework (a bundle of code that developers use to structure their application)! Depending on what framework(s) a developer uses, you might see reducers/ and actions/ folders, a mixins/ folder, or perhaps a router/ folder. Even within the scope of each framework, developers use many different file structures for different kinds of projects, so long as they promote some organized, scalable structure (see Part 5).

Some projects follow patterns[2] of application design that are independent of what frameworks you're using. Atomic Design, for example, is a principle of application design (that often goes hand in hand with component structures) that suggests that developers think about different parts of an application at different levels: atoms (tiny, simple things, like a button or some text), molecules (atoms put together, like a card), organisms (whole sections of a webpage), templates (the webpage without any data), and pages (the final webpage). Each of these five levels gets a folder. But this is once more only one possible structure--the options are infinite!
